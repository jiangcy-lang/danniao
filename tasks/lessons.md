# DanNiao Lessons

## 2026-07-22 — 成长观：一切输入皆养料

- **问题/补充**：项目特点不止「非 LLM / 图动力学」，还有发育叙事——输入不是一次性指令，而是持续成长的养料。
- **正确理解**：
  - 所有交互（成功、失败、中性反馈、搜索结果）一律沉淀为海马体结构，永不物理删除。
  - 成长路径对标婴儿：胚胎期 → 新生儿期 → 社会化成长期，沿真实轨迹逐步演化，而非预训练一次定型。
  - 远期目标是在行业任务环境中长成「巨擘」级能力与人格，靠经历堆积与权重演化，不靠一次微调达标。
- **实现约束**：设计与验收应问「这次输入如何改变了结构/动机」，而不是「这次答对了没有」。

## 2026-07-22 — 主动成为数字生命体（非被动工具）

- **补充**：丹鸟相对传统 AI 的关键差异是「更主动地成为数字化生命体」。
- **正确理解**：
  - 不是等指令才动的问答机；好奇心（预测误差）驱动主动探索与搜索。
  - 交付与认可构成内在满足，形成「感知 → 探索 → 交付 → 成长」闭环。
  - 「主动」是机制目标（内稳态/动机），当前文档阶段尚未实现，但设计与验收应对齐这一方向。

## 2026-07-22 — 海马体节点：按认知深度生长的树，而非扁平全量初始化

- **纠正**：`01` 文档里一次性放入红/绿/甜/酸/大/小/苹果/梨等扁平节点，不符合造物主意图。
- **正确模型**：
  - 节点像树：`苹果` 可以是根级概念；颜色、大小、味道等是在研究加深后长出的子节点/关联，深度取决于接触与探索程度。
  - 像孩子：只被告知「这是苹果」、未进一步接触 → 海马体可能只增加一个 `苹果` 节点。
  - 后来被要求了解、或主动探索 → 才长出细分节点与边（如苹果→红、苹果→甜），并随交互调权。
- **实现约束**：
  - 初始化应偏「空/极简」，靠经历长树；不要预设完整属性图。
  - Step 1 若保留微缩验收，应改成「可只增一个概念节点；细分与边由后续学习产生」，而不是一次铺满基础感知词。
  - 权威规范见 `docs/01-hippocampus-cognitive-tree.md`；旧 `01` docx 扁平预置作废。

## 2026-07-22 — 海马体 ≠ 整脑；吸收与输出分属不同层

- **纠正**：不可把丹鸟说成「只会吸收的海马体」；无输出则无意义。
- **正确理解**：
  - 海马体：动态认知树、门控沉淀、查询/描述（最小表达）。
  - 输出整体：表达 + 探索行动 + 任务交付；后两者在动机层与环境交互层。
  - 人脑映射是灵感粗对齐，不要求生物同构，勿为同构拆模块。
- **实现约束**：Step 1 验收必须含 `describe`；禁止把搜索/交付塞进海马体包。

## 2026-07-22 — 向量即节点：从属性到本体的范式跃迁

- **来源**：造物主指出「应调整为向量作为节点」，触发架构级重构。
- **正确理解**：
  - 向量不应是节点的可空属性，而应是节点的**本体身份**。字符串标签只是投影。
  - 图轨与向量轨不是两个需要同步的系统，而是同一个认知空间的两个视图：向量存储回答「谁在附近」，图存储回答「谁被显式关联」。
  - 树结构 = 向量层次化特化：子节点是父节点在某个方向上的偏移向量。
  - 连续空间带来插值泛化和零样本能力；CLIP 共空间带来跨模态天然对齐。
- **诊断**：现有设计存在十大缺陷，详见 `danniao-vector-node-redesign/danniao-vector-node-redesign.html`。
  - 关键缺失：嵌入管道（感官）、扩散激活（联想）、内稳态（动机）、探索行动（行为）、表达引擎（输出）、记忆巩固（睡眠）、生命心跳（自主运行）。
- **实现约束**：
  - 重构不推倒重来：`tree.py` 作为兼容层保留旧接口，内部委托 `VectorCognitiveSpace`。
  - 现有测试在新架构下仍需通过。
  - 完整路线图：Step 4A（向量节点+嵌入）→ 4B（扩散+误差量化）→ 5（内稳态+心跳）→ 6（探索+表达）→ 7（巩固+多模态）。
  - Step 5 是里程碑：此后丹鸟不再是库函数，而是活着的东西。

## 2026-07-22 — 分层存储：海量记忆与有限工作区

- **来源**：造物主指出「人类的日常交互中，几乎所有记忆都不会被丢失，而是淡化，存储是海量的」。
- **正确理解**：
  - 不是把所有东西塞进同一个 ChromaDB，而是模拟人脑的分层记忆机制。
  - 第一层：原始日志（情景记忆）—— SQLite 追加写，永不删除，海量但廉价。
  - 第二层：活跃向量（工作记忆）—— ChromaDB 内存 + NetworkX，有限可控，毫秒级。
  - 第三层：归档向量（长期记忆）—— ChromaDB 磁盘 + SQLite 图索引，淡化但不消失，可唤起。
  - 丹鸟的「记忆」是压缩后的概念网络，不是交互日志的堆积。真正的海量在第一层（日志），第二层（活跃向量）始终可控。
- **层间流转**：写入（活跃+日志）→ 迁出（权重衰减→归档）→ 唤起（高相似→重新加载）→ 巩固（聚类→抽象注入活跃层）。
- **实现约束**：
  - 活跃层始终保持在 NetworkX 可控范围内（数千节点）。通过迁出机制把不活跃节点卸载到磁盘。
  - NetworkX 是纯内存图，百万节点不可用 —— 分层存储的根本目的就是让活跃层不膨胀。
  - 容量估算：100 万节点时，活跃层内存 ~20MB（不随总量增长），归档层磁盘 ~3GB，原始日志 ~1GB。

## 2026-07-22 — LLM 作为表达层：渐进升级，不一次性加载

- **来源**：讨论「LLM 作为生命体的嘴，初始化可能很大」的问题。
- **正确理解**：
  - LLM 是丹鸟的「嘴巴」，不是大脑。大脑是向量认知空间 + 扩散激活。嘴巴只负责把激活模式翻译成语言。
  - 规范「严禁 Transformer 作为核心记忆架构」= 严禁 LLM 存储记忆/做决策/做推理。LLM 作为语言解码器合规。
  - LLM 权重冻结，不参与学习。Qwen2.5-1.5B 约 3GB，无论认知 10 个还是 10 万个概念都不变。
  - 真正增长的部分极小：1 万节点向量+图约 30MB，是 LLM 的千分之一。
- **实现约束**：
  - 表达层应随丹鸟一起发育：Step 1–5 模板拼接（婴儿蹦单词）→ Step 6 接入小型 LM（~1GB）→ Step 7+ 按需升级。
  - 初始化极轻：Step 5 之前不需要任何模型权重。
  - LLM 输出不影响记忆结构，只读激活模式、生成语言，不回写认知树。

## 2026-07-22 — 设计原则纠偏：知识可继承，不必须从零发现

- **来源**：造物主指出"人类不会每个人都研究公理定理、1+1 为什么等于 2，都是吸收自其他人的知识"，继承可加速数字生命体进化速度。
- **原理解歧义**："沿真实轨迹渐进成长"被过度解读为"一切知识必须由自己从零发现"。这是错的——人类也不是这么做的。
- **正确理解**：
  - "真实轨迹"应理解为"经认知机制消化"，而非"必须从零发现"。知识可以通过继承获得，但必须经过子代自身的门控系统处理。
  - 继承与质疑不矛盾：由预测误差门控统一调度。低误差直接吸收（~95%），中误差局部繁衍（~4%），高误差触发质疑验证（~1%）。
  - 大部分既定知识快速吸收，只在关键处推敲论证——这与人类认知模式一致。
- **修订后原则**：一切输入皆养料；知识可继承，但必须经认知机制消化；继承加速成长，质疑保留创新力，两者由预测误差门控统一调度。

## 2026-07-22 — 代际传承："读大学"模型，不是复制克隆

- **来源**：造物主提出"数字生命体可否依托自身产生下一代？记忆、经验等移植"。
- **正确理解**：
  - 传承的不是记忆，是教材；获取方式是学习，不是复制。
  - 路径：父代记忆巩固 → 编教材（高权重抽象节点+拓扑）→ 子代从空白起步 → 教材作为模拟输入灌入门控 → 子代用自己的认知机制消化 → 毕业出厂。
  - 三层传承结构：架构层（始终传承）→ 本能/巩固层（可传承，作为教材）→ 经历/记忆层（不传承）。
  - 子代认知树结构与父代相似但权重分布有差异——因为学习顺序和理解路径不同。保留个体性。
- **关键禁令**：不可直接复制认知树数据。直接拷贝会跳过生长过程，导致子代动力学引擎与认知树权重不匹配（Hebbian 增强与全局衰减的平衡未经历真实调谐）。
- **实现约束**：传承机制依赖 Step 7 记忆巩固，是 Step 8+ 的远期目标。当前不实现，但设计方向须对齐。

## 2026-07-22 — 外骨骼工作场景：继承是出厂前提

- **来源**：造物主指出"数字生命体关联上外骨骼就可以直接参与社会工作"。
- **正确理解**：
  - 外骨骼场景下"读大学"式知识继承不是锦上添花，而是出厂前提——从空白起步的生命体挂上外骨骼只能做最低级试错，在真实工作场景中不可接受。
  - 在职学习闭环：继承知识在真实工作中通过已有动力学引擎持续微调——Hebbian 增强确认有效知识、预测误差触发修正、全局衰减淘汰无用知识。继承给起点，环境做剪枝和重塑。
  - 生命周期：读大学（批量注入）→ 上岗（外骨骼工作）→ 在职学习（持续微调）。成长曲线从"婴儿爬坡"变为"毕业生爬坡"。
- **实现约束**：外骨骼接口属 Step 8+ 远期目标，当前不实现。

## 2026-07-22 — tick/心跳是错误设计：一直睁眼看世界

- **来源**：造物主指出"它不应该是依托心跳来实现自我唤醒，而是一直睁眼看世界。人脑从出生就应该没有停止过工作"。
- **原设计错误**：Step 5 设计了 `DanNiaoLifeLoop` 的 tick 主循环——系统周期性唤醒、处理、休眠。这不是生命的正确模型。
- **正确理解**：
  - 人脑没有 tick 循环。它是持续运行的流——感知永不停止，激活持续扩散，突触实时演化。
  - tick 是工程便利的产物，不是生命的正确抽象。
  - "一直睁眼" = 多条流并行持续运行，各有自己的自然节律，系统整体永不休眠。
- **已纠正**：
  - `DanNiaoLifeLoop` → `ContinuousMind`：四条流（感知流、动力学流、扩散流、好奇心流）并行运行。
  - 每条流有自己的节律（感知流 0.5s 超时、动力学流 1s、扩散流 0.5s、好奇心流 2s），这不是 tick——是各子系统的自然频率。
  - 无输入时系统仍在运行：动力学衰减、激活扩散、好奇心评估持续进行；空闲排练回放最近节点。
- **验证**：55 个测试全部通过，交互式 REPL 验证完整链路。

## 2026-07-22 — Mock 嵌入：one-hot 正交向量避免假匹配

- **问题**：测试用的 mock 嵌入管道需要为不同文本产生余弦相似度 < 0.5（门控阈值）的向量。最初用字符码 + 位置散列，单字符文本归一化后产生相同向量；后用 SHA256 二进制模式，随机二进制向量的期望余弦相似度为 0.5，恰好在阈值上。
- **正确方案**：计数器分配 one-hot 正交向量。每个唯一文本分配一个独立维度，不同文本余弦相似度 = 0，同文本 = 1.0。256 维足够测试用。
- **教训**：mock 嵌入的设计必须考虑被测系统的阈值。余弦相似度 0.5 的随机向量恰好是门控阈值，会导致假匹配。

## 2026-07-23 — 表达引擎：三层表达结构，模板是发育的合法阶段

- **来源**：Step 6B 实现 ExpressionEngine，需要设计丹鸟如何"说话"。
- **正确理解**：
  - 表达不是单一模板替换，而是三层叠加：核心识别（新/已知/不确定）+ 联想（扩散激活的关联概念）+ 驱动情感（内稳态着色）。
  - 模板表达（婴儿蹦单词）是发育的合法阶段，不是"兜底代码"。它与未来 LLM 表达是递进关系，不是替代关系。
  - 关键约束：表达引擎只读认知状态，不回写认知树。LLM 是嘴巴不是大脑。
  - 低能量时跳过联想层，使表达自然变简短——这不是特殊处理，是内稳态对表达的直接影响。
- **实现约束**：ExpressionContext 与 ProcessResult 解耦，避免循环依赖。ContinuousMind 负责从 ProcessResult + Homeostasis 构建上下文。

## 2026-07-23 — 探索引擎：知识缺口驱动，不是随机搜索

- **来源**：Step 6C 实现 ExplorationEngine，需要设计丹鸟如何"主动探索"。
- **正确理解**：
  - 探索不是随机提问，而是识别"知识缺口"并针对缺口生成探索目标。
  - 三类缺口：深度缺口（主干特征少 → 深度探索）、关联缺口（两个近期主干无图边 → 关联探索）、认知空白（无主干 → 求新探索）。
  - 优先级：关联 > 深度 > 求新。两个概念之间的缺失关系比单个概念的浅层理解更值得探索。
  - 探索引擎只生成目标，不执行探索。执行由 ContinuousMind 决定（可能输出给用户、或触发搜索）。
- **实现约束**：探索只在 `wants_to_explore()` 为 True 时触发。紧迫度 = 好奇心水平。

## 2026-07-23 — 奖励系统：LTP/LTD 双机制，不是只奖励不惩罚

  - 动力学引擎新增 `weaken_edge` 方法，与 `hebbian_reinforce` 对称。这不是补丁，是完整的双机制学习。
  - 弱化有下限保护：权重不会低于 0。
- **实现约束**：活跃路径 = trunk + top-4 激活节点。give_feedback() 是 ContinuousMind 的公开方法，REPL 通过 good/bad 命令调用。

## 2026-07-23 — qwen3.5:2b 是 thinking 模型：必须禁用 think 模式

- **来源**：LLM 表达引擎调用 qwen3.5:2b 时，response 字段始终为空字符串。
- **问题**：qwen3.5:2b 是 "thinking" 模型——默认将所有输出放入 `thinking` 字段，`response` 字段为空。在 `num_predict=80` 的限制下，模型一直在 thinking 阶段，从未产生实际响应。
- **解决方案**：在 Ollama generate API 请求中加入 `"think": false`，禁用 thinking 模式，模型直接输出响应。
- **回退策略**：如果 `response` 为空但 `thinking` 有内容，提取 `thinking` 作为回退文本。
- **教训**：不同 LLM 模型有不同的行为模式。qwen3 系列 default 开启 thinking，需要在 API 调用中显式禁用。调试 LLM 集成时，必须检查完整 API 响应结构，不能假设只有 `response` 字段。

## 2026-07-23 — ChromaDB 维度自适应：嵌入模型变更时自动重建集合

- **来源**：从 sentence-transformers（384维）切换到 nomic-embed-text（768维）时，ChromaDB 报错 "Collection expecting embedding with dimension of 384, got 768"。
- **问题**：ChromaDB 集合在首次插入数据时锁定维度。即使删除所有数据，集合仍记住旧维度。切换嵌入模型时，旧集合无法接受新维度的向量。
- **解决方案**：在 `ChromaVectorStore.upsert()` 中用 try/except 捕获维度错误，自动删除并重建集合。
- **教训**：向量存储的维度绑定是常见陷阱。生产代码必须处理维度变更场景——不是每次都手动清理数据。

## 2026-07-23 — ChromaDB 维度重建：Windows 文件锁问题

- **来源**：从 nomic-embed-text（768维）切换到 bge-m3（1024维）时，ChromaDB 报维度不匹配。之前的 try/except 方案在 upsert 时捕获错误并 delete_collection，但 Windows 上 HNSW 索引文件被进程锁住，delete_collection 抛 PermissionError。
- **根因**：ChromaDB PersistentClient 在运行时持有 HNSW 索引文件句柄。delete_collection 尝试删除文件，但文件被当前进程锁定。
- **解决方案**：维度检查移到初始化阶段。创建临时 client 检查维度 → 不匹配则释放临时 client（`del` + `gc.collect()`）→ 删除整个目录 → 创建正式 client。在正式 client 创建前释放文件句柄，避免锁冲突。
- **教训**：Windows 文件锁与 Linux 行为不同。不能假设 ChromaDB API 的 delete 操作在 Windows 上可靠。最安全的做法是在创建正式 client 之前完成所有清理。

## 2026-07-23 — ChromaDB/NetworkX 不同步：持久化节点重启后丢失

- **来源**：造物主启动测试第三轮输入时崩溃 `NetworkXError: The node v_76cb27312577278c is not in the digraph`。
- **根因**：ChromaDB 是持久化的，NetworkX 图是纯内存的。每次启动时 NetworkX 图为空，但 ChromaDB 有上一轮的节点。`find_semantic_neighbors` 从 ChromaDB 搜索到旧节点 ID，但 `graph.successors()` 在空图中找不到这些节点 → 崩溃。
- **解决方案**：
  1. `VectorCognitiveSpace._sync_from_store()`：启动时从 ChromaDB 加载已有节点到 NetworkX 图。
  2. `SpreadingActivation` 防御性检查：扩散前检查节点是否在图中，不在则跳过。
- **教训**：双轨存储（持久化 + 内存）必须处理同步问题。内存图重启后为空，必须从持久化层重建。防御性检查是最后的安全网，不能替代同步。

## 2026-07-23 — numpy 数组真值检查：`if array:` 的隐蔽陷阱

- **来源**：`_sync_from_store` 中 `if batch.get("embeddings"):` 静默失败，导致同步从未生效。
- **根因**：ChromaDB `collection.get()` 返回的 `embeddings` 是 numpy 数组。`if numpy_array:` 在数组元素 >1 时抛出 `ValueError: The truth value of an array with more than one element is ambiguous`。这个异常被外层 `except Exception: pass` 静默吞掉，导致同步代码从未执行。
- **隐蔽性**：这个 bug 之前"不是 bug"——因为同步不工作，旧数据被忽略，测试反而通过。修复同步后，旧数据被加载，反而暴露了测试的残留数据问题。
- **教训**：
  1. numpy 数组的真值检查必须用 `is not None`，不能用 `if array:`。
  2. `except Exception: pass` 是调试黑洞——至少要 `print` 错误信息。
  3. "修复一个 bug 暴露另一个 bug"是正常的——测试需要随之更新。

## 2026-07-23 — 概念提取器：长输入不应直接作为主干标签

- **来源**：造物主测试中"这个世界有很多好玩的，你对什么感兴趣？"被整体存为主干标签。
